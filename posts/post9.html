# Agent工作流

从Vibe Coding到Agent工作流：40天没写代码，产出却超过去一年半

📅 2026-05-19 · 🕒 8分钟阅读

很久没更新了。虽然也没人看，但是还是总结一下做了什么。

开始使用agent并开发agent后，进化速度已经是指数级上升了。不论是学习能力，工程能力亦或者认知水平，都发生了质变。我已经40天没有手写过代码了，但是项目产出超过了前面一年半的总和。

实际上，内容产出和我的自身代码水平没有关系。通过vibe coding，我需要做的是产出idea、提供token并训练agent，以及开发prompt和插件就行。例如Redis架构以及OA系统的框架流，并不是我学会了框架设计然后开发，而是AI来设计并开发，我在review和push的时候去学习和理解。

由于自身做了快半年外包，又没机会进大厂，还是想去初创类小厂闯一闯的，尽管到现在拿到的都是黑奴offer。但凡今年是大三，已经拿下中厂了我说实话。

今天这篇post还是讲一下关于这两天搓的一套agent工作链。

Agent工作流：核心不是Prompt，是链路规范

实际上，除去应用场景和prompt优化，agent本身最重要的就是插件的规范链路以及适配。例如，在链路失败回滚过程中，我们需要严格规范每处回滚的节点和边界条件。如果回滚错乱，会导致推理爆炸、边界报错甚至是安全风险。

基于这个认知，我设计了一套通用的Agent工作流框架，并已经在 ram-agent 项目中实现（基于Matt Pocock的微技能架构）。下面是链路设计（分两阶段展示，避免单图过长）：

ram-agent：微技能架构的落地实践

基于上述工作流思想，我开源了 Ram-Agent 套件：https://github.com/RaM-bugourp/Ram_agents。这是一个面向全栈开发的微技能Agent套件，核心理念是：一个技能 = 一个职责，每个子技能专注单一关注点，通过路由器串联组合。

链路失败回滚协议

最新提交中，我专门添加了 common/ERROR-PROTOCOL.md，规范了跨技能的错误传递和回滚边界。

flowchart TD Error[子技能执行错误] --> Capture[捕获错误类型与上下文] Capture --> Check{错误级别} Check -->|可恢复| Retry[有限重试 ≤3次] Retry --> RetrySuccess{成功?} RetrySuccess -->|是| Resume[继续原链路] RetrySuccess -->|否| Rollback Check -->|不可恢复| Rollback[触发回滚] Rollback --> Undo[执行补偿操作 状态还原] Undo --> Boundary[回到上一个安全边界节点] Boundary --> Fallback{兜底路径存在?} Fallback -->|是| FallbackExec[执行兜底逻辑] Fallback -->|否| Escalate[异常升级 → 人工介入] style Error fill:#f8d7da,stroke:#dc3545,color:#721c24 style Rollback fill:#f8d7da,stroke:#dc3545,color:#721c24 style FallbackExec fill:#fff3e0,stroke:#e6a017,color:#856404 style Escalate fill:#e2e3e5,stroke:#6c757d,color:#383d41
📌 图3：链路失败回滚协议 — 保证Agent执行稳定性

当时我搓这个Agent工作链时，踩过最大的坑就是回滚错乱。Agent推理到一半，因为一个边界条件没处理好，整个上下文崩了，然后模型开始幻觉补全，最后输出了一堆不可控的内容。所以后来我把回滚测试/兜底路径作为一等公民纳入框架。

补充：工作流还缺什么？

在社区讨论中，我也收到了对这套框架的补充建议。除了当前已实现的节点外，还应该考虑：

  • 结果校验 / 断言环节：在「具体工作流」之后增加任务完成度检查
  • 异常降级 / 兜底路径：与「回滚测试」并列，提供默认安全路径
  • 反馈闭环 / 学习入口：输出review之后增加规则更新节点

部署与使用

将 ram-agent/ 目录放置到 WorkBuddy 技能目录下：

# 用户级安装（所有项目可用）
~/.workbuddy/skills/ram-agent/

# 项目级安装（仅当前项目）
{project}/.workbuddy/skills/ram-agent/

触发方式：@ram 或 ram + 任务描述。也可直接触发子技能：@ram-diagnose、@ram-architect。

🔗 查看 GitHub 仓库：Ram_agents →

总结

Agent工作流的核心不在于单个Prompt写得多么精美，而在于链路的规范性和回滚机制的完备性。当你把流程拆解成可插拔、可回滚、可观测的节点序列，AI的产出稳定性和可控性会上一个台阶。
