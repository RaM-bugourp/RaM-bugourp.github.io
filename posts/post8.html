# 后端规范

日志：后端开发的另一只眼

📅 2026-04-01 · 🕒 8分钟阅读

一、还有Call back

在上一次的哈希存储文章里，代码块中藏着这么一行：

logging.info('The salt has generated')

看起来轻飘飘，像是一句自言自语。但对于写这行代码的人来说，它可能是未来某个凌晨三点、被甲方电话吵醒时，唯一能救命的东西。

这就是日志。 绝大多数新手开发者刚开始不会考虑它——因为初期的甲方成熟度不够、项目规模也小，出bug直接翻代码看一眼就行。但项目一旦跑起来，用户多起来，或者甲方开始认真运营，“查日志”就成了唯一的破案手段。

二、除非你是外包

假设你写了一个哈希存储+验证的功能，上线后用户突然反馈“密码正确但登不上”。你盯着代码看了半天，逻辑明明没问题：

  • 注册时存了 salt 和哈希
  • 登录时取出 salt 重新计算
  • 两个哈希值一模一样

可用户就是登不上去。问题出在哪？

  • 数据库里的数据同步出错了？
  • 前端传过来的密码被截断了？
  • 后端收到的密码编码不对？
  • 验证函数根本没被调用？

如果没有日志，你只能靠猜，或者让用户陪你一起“重现现场”。但有了日志，情况完全不同。

🐛 真实场景复现（用户反馈登录失败）
INFO 2026-04-01 21:33:12 - 收到登录请求 | 用户名: zhangfei | IP: 221.3.xxx.xx
DEBUG 2026-04-01 21:33:12 - 从数据库查询用户: zhangfei
DEBUG 2026-04-01 21:33:12 - 查询结果: 用户存在, salt=7f2a3b..., stored_hash=8d96...
DEBUG 2026-04-01 21:33:12 - 对输入密码重新哈希: 输入长度=8, 新哈希=9a3f...
WARNING 2026-04-01 21:33:12 - 哈希比对失败 | 数据库哈希: 8d96... | 计算哈希: 9a3f...
WARNING 2026-04-01 21:33:12 - 登录失败: 密码错误 | 用户名: zhangfei

看到这条日志，你瞬间明白：密码确实错了，不是代码bug，也不是数据库问题。直接告诉用户“密码错误”就行，不用熬夜翻代码。

但假如日志里出现了另一组信息：

ERROR 2026-04-01 21:35:01 - 数据库查询异常: (connector, 'MySQL server has gone away')
ERROR 2026-04-01 21:35:01 - 函数: verify_password() 执行失败 | traceback: ...

你就知道——根本就不是密码问题，是数据库连接断了。日志帮你精准定位到哪一层崩了，而不是让你在“前端”和“后端”之间反复横跳。

三、打个大西瓜哦

很多新手把所有日志都打成 INFO，或者用 print 替代，这是典型的“自毁长城”。Python logging 模块定义了五个标准级别，用对了事半功倍：

级别用途示例场景
DEBUG调试信息，开发/测试环境用变量值、SQL语句、缓存命中
INFO正常流程关键节点请求开始/结束、用户登录、订单创建
WARNING可恢复的异常密码错误、参数缺失、重试机制触发
ERROR功能受损但程序未崩溃数据库连接失败、API调用超时
CRITICAL严重错误，程序可能退出磁盘写满、核心服务宕机

💡 实战技巧：开发环境用 DEBUG 级别，生产环境用 INFO 或 WARNING，避免日志刷爆磁盘。同时可以给不同的模块设置独立的日志文件，例如 auth.log、db.log。

四、日志到底在记什么？

很多人以为日志就是 console.log，那顶多算“临时打印”。真正的项目日志至少应该包含：

  • 时间戳：什么时候发生
  • 级别：DEBUG / INFO / WARNING / ERROR / CRITICAL
  • 位置：哪个文件、哪一行、哪个函数
  • 上下文：用户名、IP、请求ID、关键参数
  • 错误堆栈：异常发生的完整链条

上面那段哈希代码里，我们其实只打了“Salt generated”这一条。对于一个正式项目，至少要这么打：

def new_hash_password(password):
    logging.info(f"开始哈希处理 | 密码长度: {len(password)}")
    salt = os.urandom(32)
    logging.debug(f"盐已生成 | 盐长度: {len(salt)}")
    pwd_hash = hmac.new(salt, password.encode('utf-8'), hashlib.sha256).hexdigest()
    logging.info(f"哈希完成 | 结果长度: {len(pwd_hash)}")
    return f"{salt.hex()}:{pwd_hash}"
def verify_password(input_password, stored_password):
    logging.info("开始密码验证")
    try:
        salt_hex, stored_hash = stored_password.split(':')
        logging.debug(f"盐解析成功 | 盐长度: {len(salt_hex)}")
        salt = bytes.fromhex(salt_hex)
        new_hash = hmac.new(salt, input_password.encode('utf-8'), hashlib.sha256).hexdigest()
        result = (stored_hash == new_hash)
        logging.info(f"验证完成 | 结果: {result}")
        return result
    except Exception as e:
        logging.error(f"验证过程异常 | 错误: {str(e)}", exc_info=True)
        return False

这样哪怕你在睡觉，第二天起来翻日志，也能一眼看出是盐格式坏了，还是输入编码有问题，还是数据库存的数据缺斤少两。

五、你也曾为删了浏览记录懊悔吗

📌 飞机黑匣子记录飞行数据，日志记录程序运行数据。
没有它，坠机（崩服）后你只能靠猜。有它，你甚至能还原事故发生前 10 秒的每一条指令。

对于接单开发来说，日志还有一个隐藏价值：向甲方证明你在干活，也证明问题出在谁身上。当甲方说“你的代码有问题”时，你甩出一段带时间戳的日志，清晰显示是服务器磁盘满了 / 第三方接口超时 / 他们的人改错了配置，对方哑口无言，你也不用背锅。

六、所以我喜欢开无痕模式

生产环境跑久了，日志文件会越来越大。Python 的 RotatingFileHandler 可以按大小自动切分：

from logging.handlers import RotatingFileHandler

handler = RotatingFileHandler(
    'app.log',
    maxBytes=10 * 1024 * 1024,   # 10MB
    backupCount=5                 # 保留5个备份
)

另外，千万别把密码、token 之类的敏感信息直接打进日志。可以用脱敏工具或者手动截断：

def mask_sensitive(data):
    if 'password' in data:
        data['password'] = '***'
    return data

七、我趣，不早打

哈希存储是数据泄露的最后一道防线，而日志是你排查问题的第一把手术刀。

初学时你觉得它多余，是因为你还没碰到真正让你手足无措的bug。一旦遇到，你会恨不得穿越回去，在每个关键节点塞满日志。

所以，下次写代码时——

  • 函数入口打一条 INFO
  • 关键分支打一条 DEBUG
  • 异常捕获打一条 ERROR（带上堆栈）
  • 外部调用（数据库、API）前后都打一条

📎 附：文首那段哈希存储的完整日志配置（可直接复用）

import logging
import os

log_dir = 'log'
if not os.path.exists(log_dir):
    os.makedirs(log_dir)

logging.basicConfig(
    filename='log/info.log',
    level=logging.INFO,
    format='%(asctime)s - [%(levelname)s] - %(filename)s:%(lineno)d - %(funcName)s() - %(message)s',
    datefmt='%Y-%m-%d %H:%M:%S',
)
