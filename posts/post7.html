# 后端安全

哈希？数据泄露的最后一道防线？

📅 2026-03-31 · 🕒 8分钟阅读

一、要对甲方负责哦 魂淡

当我们开始接触后端开发，不可避免要面对用户的敏感数据存储——密码、账户余额、API密钥……恶意渗透攻击下，数据库暴露是很难完全避免的。所以我们不仅要从前端到数据库调用层层设防，对于数据本身，也要做好被窃取的准备。

于是，在后端开发中，敏感数据哈希存储成为了必须进行的基本安全规范。

你能从一盘番茄炒蛋里，还原出当初放了多少克盐吗？不能。但只要盐差了一粒，整盘菜的味道就完全不同。这正好对应了哈希的两个核心特性：单向性（不可逆）和雪崩效应（输入微小变化，输出天差地别）。

这里“盐”——也就是我们常说的 salt，就是哈希过程中生成的随机密钥。即使两个人密码相同，盐不一样，最终存进数据库的哈希值也完全不同。

三、不要抄我代码，自己写

当后端收到用户密码，不会直接存明文。而是随机生成一个 salt，然后用它对密码做哈希，最后把 salt:哈希值 一起存入数据库。

import os
import hashlib
import hmac
import logging

def new_hash_password(password):
    # 生成32位随机盐
    salt = os.urandom(32)
    logging.info('Salt generated')
    
    # hmac.new(密钥, 消息, 摘要算法)
    pwd_hash = hmac.new(salt, password.encode('utf-8'), hashlib.sha256).hexdigest()
    
    # 返回"盐:哈希值"的格式，方便存入数据库
    return f"{salt.hex()}:{pwd_hash}"

存入数据库的一条记录大概长这样：
a3f2c1...:8d969eef6ecad3c29a3a629280e686cf...
冒号前面是盐的十六进制，后面是哈希结果。

四、那登录的时候怎么验证？

有人可能会问：“我自己都不知道哈希值对应的明文是什么，怎么判断用户输对了密码？”

这就是我们保存 salt 的原因。我们无法从哈希反推出原始密码，但我们可以重复“加盐 → 哈希”的过程。

def verify_password(input_password, stored_password):
    # 从数据库里存的字符串中分离出盐和哈希值
    salt_hex, stored_hash = stored_password.split(':')
    salt = bytes.fromhex(salt_hex)  # 转回二进制盐
    
    # 用同样的盐对用户这次输入的密码做哈希
    new_hash = hmac.new(salt, input_password.encode('utf-8'), hashlib.sha256).hexdigest()
    
    # 对比两个哈希值是否相同
    if stored_hash == new_hash:
        return True
    else:
        return False

验证流程就是：

  • 用户登录时输入密码 → 取出数据库中存的 salt:哈希值
  • 用相同的 salt 对输入密码重新哈希 → 得到一个新的哈希值
  • 如果新哈希值和数据库里存的哈希值相等 → 密码正确，允许登录

五、为什么能成为规范

📌 一句话总结
由于哈希的单向性，数据库中的密码不用于“记录”，而仅仅用于“验证后续登录”。即使黑客把整张表拖走，也拿不到任何原始密码。从根源上避免了数据泄露。

这就是为什么现代系统几乎都用 bcrypt、Argon2 或我们上面这种 HMAC-SHA256 + 随机盐的方案来存储密码。它把“保管密码”这件事，变成了“保管一种不可逆的指纹”。

而对于我们开发者来说，实现这套机制并不复杂，但它决定了你的用户数据是不是裸奔。

六、我真得教你当黑客了

后端开发的世界里，安全永远不是“可选”。不止于哈希存储，往后还有 SQL 注入、XSS、CSRF、权限控制……但至少，先把密码这件事做好，是对用户最基本的尊重。
